# SO(16) Matrix Approach: Design Document

Date: 2026-03-14
Status: Research complete, prototype written, compilation pending

## Problem Statement

The flat 120-field SO(16) type (`so16_grand.lean`) compiles but produces a 2.2 GB
`.olean` file. This exceeds 32-bit `stat` limits, causing downstream files
(`so14_so16_liehom.lean`) to fail on import. We need an alternative representation
that avoids the .olean size explosion.

## Mathlib Matrix Lie Algebra Infrastructure

### Key discovery: mathlib already defines so(n) as a LieSubalgebra

The exact definition lives in `Mathlib.Algebra.Lie.Classical`:

```lean
namespace LieAlgebra.Orthogonal

def so [Fintype n] : LieSubalgebra R (Matrix n n R) :=
  skewAdjointMatricesLieSubalgebra (1 : Matrix n n R)

theorem mem_so [Fintype n] (A : Matrix n n R) : A ∈ so n R ↔ Aᵀ = -A
```

So `LieAlgebra.Orthogonal.so (Fin 16) ℝ` is **already a valid LieSubalgebra** of
`Matrix (Fin 16) (Fin 16) ℝ`, and it automatically inherits:
- `LieRing` (from `LieSubalgebra.lieRing`)
- `LieAlgebra ℝ` (from `LieSubalgebra.lieAlgebra`)

No manual bracket definition needed. No Jacobi proof needed. No 120-field structure.

### How it works: the instance chain

1. `Matrix (Fin 16) (Fin 16) ℝ` is a `Ring` (matrix multiplication)
2. Any `Ring` is a `LieRing` via `LieRing.ofAssociativeRing`:
   `[A, B] = A * B - B * A` (the commutator)
3. Any `Algebra R A` with `A` a ring is a `LieAlgebra R A`
4. `skewAdjointMatricesLieSubalgebra (1 : Matrix n n ℝ)` = the set
   `{A | Aᵀ * 1 = 1 * (-A)}`, which simplifies to `{A | Aᵀ = -A}`
5. This is closed under the Lie bracket (proved in `Mathlib.Algebra.Lie.SkewAdjoint`)
6. Therefore it is a `LieSubalgebra`, which inherits `LieRing` and `LieAlgebra ℝ`

### Key mathlib files

| File | What it provides |
|------|-----------------|
| `Mathlib.Algebra.Lie.OfAssociative` | `[A,B] = A*B - B*A` for any ring; `LieAlgebra.ofAssociativeAlgebra` |
| `Mathlib.Algebra.Lie.Matrix` | `lieEquivMatrix'` (endomorphisms iso matrices as Lie algebras) |
| `Mathlib.Algebra.Lie.SkewAdjoint` | `skewAdjointMatricesLieSubalgebra J` closed under bracket |
| `Mathlib.Algebra.Lie.Classical` | `LieAlgebra.Orthogonal.so n R` = skew-adjoint w.r.t. identity |
| `Mathlib.Algebra.Lie.Subalgebra` | `LieSubalgebra.lieRing`, `.lieAlgebra`, `.incl` |

### The bracket

For matrices, `[A, B] = A * B - B * A`. This is the standard matrix commutator.
For antisymmetric matrices (elements of so(n)), this commutator stays antisymmetric:
if `Aᵀ = -A` and `Bᵀ = -B`, then `(AB - BA)ᵀ = BᵀAᵀ - AᵀBᵀ = BA - AB = -(AB - BA)`.

### Indefinite signature

Mathlib also provides `LieAlgebra.Orthogonal.so' p q R`, which is the indefinite
orthogonal algebra so(p,q) -- skew-adjoint with respect to the indefinite diagonal
`diag(1,...,1,-1,...,-1)`. There is a proven equivalence:
```lean
noncomputable def soIndefiniteEquiv {i : R} (hi : i * i = -1) :
  so' p q R ≃ₗ⁅R⁆ so (p ⊕ q) R
```
This could be relevant if we later need so(11,3) or so(14,2).

## Design for so16_matrix.lean

### Approach: Use mathlib's `so` directly as a type abbreviation

```lean
import Mathlib.Algebra.Lie.Classical

abbrev SO16M := LieAlgebra.Orthogonal.so (Fin 16) ℝ
```

This immediately gives us:
- `LieRing SO16M` (inherited from `LieSubalgebra`)
- `LieAlgebra ℝ SO16M` (inherited from `LieSubalgebra`)
- `SO16M.incl : SO16M →ₗ⁅ℝ⁆ Matrix (Fin 16) (Fin 16) ℝ` (the inclusion)

### Dimension

The dimension of so(n) is n(n-1)/2 = 16*15/2 = 120. To prove this in Lean:
- We need `Module.finrank ℝ SO16M = 120`
- This requires showing that the submodule of antisymmetric 16x16 matrices has
  rank 120 over ℝ
- This is likely provable but may require nontrivial linear algebra (a basis
  construction for skew-symmetric matrices)
- We can STATE this and possibly prove it, but it is not required for the
  LieHom chain

### Embedding SO14 into SO16 via matrices

**Option A: Matrix-to-matrix subalgebra inclusion**

Define SO14M similarly and use the natural block inclusion:
```lean
abbrev SO14M := LieAlgebra.Orthogonal.so (Fin 14) ℝ

def so14_block_embed (A : Matrix (Fin 14) (Fin 14) ℝ) : Matrix (Fin 16) (Fin 16) ℝ :=
  Matrix.fromBlocks A 0 0 0  -- after reindexing Fin 16 ≃ Fin 14 ⊕ Fin 2
```

The key challenge: proving this preserves skew-adjointness and the bracket.
The block-diagonal embedding of a skew-adjoint matrix into a larger matrix
(padding with zeros) is skew-adjoint. The bracket is preserved because the
off-diagonal blocks are zero.

**Option B: SO14_flat → SO16M via an explicit LieHom**

Map each flat SO14 element to an antisymmetric 16x16 matrix:
```lean
def so14_to_matrix (x : SO14) : Matrix (Fin 16) (Fin 16) ℝ :=
  -- place x.l12 at position (0,1), -x.l12 at (1,0), etc.
  -- zero out rows/columns 14,15
```

Then prove:
1. The output is skew-adjoint (so it lands in SO16M)
2. It preserves addition and scalar multiplication
3. It preserves the bracket

This is the "bridge" approach: connect the existing flat type to the matrix type.

**Recommended: Option A for SO16M standalone, then bridge SO14_flat → SO16M**

### Comparison: flat vs matrix representation

| Aspect | Flat (120 fields) | Matrix (Fin 16) |
|--------|------------------|----------------|
| .olean size | 2.2 GB (broken) | Expected small (uses mathlib types) |
| Jacobi proof | Manual, huge | Free (inherited from associative ring) |
| LieRing instance | Manual, 200M heartbeats | Free (inherited from LieSubalgebra) |
| Bracket definition | 14,400 terms | Free (matrix commutator) |
| Element access | Direct field access | Matrix indexing |
| Compatibility | Works with existing SO14 LieHom | Needs bridge map |
| ext lemma | Auto from @[ext] | Subtype.ext + matrix ext |

### Bridge strategy: connecting flat SO14 to matrix SO16

The existing chain is:
```
SU5C →ₗ⁅ℝ⁆ SO10 →ₗ⁅ℝ⁆ SO14 →ₗ⁅ℝ⁆ SO16_flat
```

The new chain would be:
```
SU5C →ₗ⁅ℝ⁆ SO10 →ₗ⁅ℝ⁆ SO14 →ₗ⁅ℝ⁆ SO16M
```

We need to replace only the last arrow. The map SO14 → SO16M sends each generator
L_{ij} (with i,j in {1,...,14}) to the corresponding antisymmetric matrix in
`Matrix (Fin 16) (Fin 16) ℝ`, with zeros in rows/columns 14 and 15 (0-indexed).

The bracket-preservation proof should be doable by `simp` with matrix lemmas,
since both brackets are ultimately matrix commutators. The flat SO14 bracket was
defined to match the matrix commutator formula.

### What about E_8?

If the project eventually needs E_8 (248 generators), the matrix approach won't
directly help -- E_8 is exceptional, not a classical matrix Lie algebra in the
standard sense. However:
- E_8 can be constructed as a Lie subalgebra of `gl(248)` via the adjoint
  representation
- The so(16) subalgebra of E_8 (the 120-dimensional part) would use this
  matrix construction
- The spinor part (128-dimensional) requires additional structure

## Risks and Mitigations

### Risk 1: Definitional unfolding performance
Matrix operations over `Fin 16` involve `Finset.sum` over 16 elements. Lean's
kernel may struggle to evaluate these symbolically.

**Mitigation**: Avoid `decide` and `native_decide` for large finite computations.
Use `simp` with appropriate lemmas instead.

### Risk 2: Bridge map complexity
The SO14_flat → SO16M map has 91 components. Proving bracket preservation requires
showing that the flat bracket formula matches the matrix commutator for all 91
output positions.

**Mitigation**: This is fundamentally the same proof as the existing
`so10_toSO14_lie` theorem (which already works with 128M heartbeats). The matrix
side may actually be simpler because `simp` can reason about matrix multiplication
directly.

### Risk 3: Downstream compatibility
Files that import `so16_grand.lean` will need to be updated to use `so16_matrix.lean`.

**Mitigation**: Currently only `so14_so16_liehom.lean` imports `so16_grand.lean`.
That file would be replaced by the new bridge map.

## Final Design (as implemented in prototype)

The prototype `src/lean_proofs/clifford/so16_matrix.lean` uses TWO index types:

1. `SO16M := LieAlgebra.Orthogonal.so (Fin 16) R` -- the canonical type
2. `SO16Block := LieAlgebra.Orthogonal.so (Fin 14 + Fin 2) R` -- for block decomposition

Connected by `Matrix.reindexLieEquiv finSumFinEquiv` at the full matrix level.

The block embedding uses `Matrix.fromBlocks`:
```lean
def blockEmbed (A : Matrix (Fin 14) (Fin 14) R) :
    Matrix (Fin 14 + Fin 2) (Fin 14 + Fin 2) R :=
  Matrix.fromBlocks A 0 0 0
```

This gives very clean proofs:
- Skew-symmetry preservation: 3-line proof via `fromBlocks_transpose` + `fromBlocks_neg`
- No nested `dite` case analysis needed
